Peal Network · encrypted mempool

An encrypted mempool that opens itself, end to end

A public mempool leaks your trade before it executes, so a searcher reads it and sandwiches you. Peal seals every order to a committee, orders the ciphertexts blind, and opens the whole batch at once on a cue. This is how the live demo works, with the real numbers from the running stack on Tempo testnet.

The problem: a readable mempool is a free option for searchers

On a normal chain your swap waits in the public mempool in plain sight. Anyone watching, including automated searchers, can read the amount, the direction, and the price you are willing to accept, all before it executes. That visibility is the whole attack:

1

your order is public

the searcher sees your swap and your slippage floor.

2

the searcher front-runs

it buys just ahead of you, pushing the pool price against you.

3

you fill worse, it sells back

your swap executes at the worse price, the searcher sells into it, and the difference (sized to your own slippage limit) is its profit.

The searcher's edge is not compute, it is information. Peal removes the information. The order travels sealed, builders order ciphertexts they cannot read, and by the time anything is readable the ordering is already fixed. The searcher's problem is not made harder, it is made empty.

The architecture

The demo runs a full pipeline. The seal-and-reveal core is the Peal network (a coordinator, a committee of operator nodes, and a browser SDK). Around it sit the pieces that make the mempool comparison real on-chain: two AMM pools, a public-builder contract, a Peal-mempool contract, and three off-chain agents.

The contracts model the two worlds with the same adversary:

  • SwapPool: a constant-product pool (x*y=k, 0.3% fee) whose swap is gated to one operator, the builder allowed to move it. Deployed twice, one per lane.
  • PublicBuilder: models an unprotected mempool. submitOrder defers execution and emits the order in cleartext, so the searcher sees amount, direction, and floor. buildBlock([frontRun, victim, backRun]) lets it execute its chosen ordering, a real sandwich with real extraction.
  • PealMempool: commitSealed(conditionId, ctHash) emits only the hash, so the searcher has nothing to wrap. executeBatch(conditionId, orders, merkleRoot) is coordinator-only, recomputes the merkle root over the revealed order payloads using the sha256 precompile, and swaps in committed order only if the root matches.

Settlement runs as a small TypeScript service (the settler) that watches the coordinator's reveal API, rather than inside the Rust coordinator. That keeps the shipped devnet untouched and all chain logic in one place.

How batched threshold encryption does the work, step by step

Peal is built on batched threshold encryption (the commonware simple-bte construction, FO variant). The one-sentence version: encrypt each transaction to a committee key, and when a cue fires the operators each return one tiny value that together open the entire batch at once. Here is the full path an order takes.

01

seal, on the user's device

The browser SDK encrypts the order in wasm to the committee's public key. The output is a capsule: a 48-byte KEM header (one group element) plus an opaque payload. The amount, direction, and token never leave the device in the clear. The capsule's ctHash = sha256(ciphertext) is its fingerprint.

02

attach a cue

The order is sealed to a condition: for the mempool demo, a time cue five seconds out. The condition is what decides when the batch may open. Nothing opens before it fires.

03

commit the hash on-chain

The relayer calls commitSealed(conditionId, ctHash). Only the hash goes on-chain, so the position is anchored before anyone can read the order. There is nothing for a searcher to front-run.

04

freeze and pad to a fixed batch

At the cue the coordinator freezes the batch to exactly 64 slots. Real ciphertexts sort by ctHash; dummy padding fills the tail. Positions are a pure function of the real set, so no observer can tell how many real orders are inside or which slot is yours.

05

pre-decrypt, pipelined

The heavy per-batch cryptography (cross terms) is precomputed at freeze while shares are still in flight, so it never sits on the user-visible path. The coordinator ticks every 500ms.

06

operators return shares

Each operator returns exactly one 48-byte share for the whole batch, not one per transaction. Every share is checked with a public pairing equation before it counts. Any three of the five are enough; a rejected share never counts toward the threshold.

07

combine and reveal

Three valid shares combine and the entire batch decrypts together. In a live run this finalize step took 90 ms. The coordinator publishes the plaintexts and a merkle root over (position, payload) for every slot, padding included.

08

settle in fixed order

The settler reads the revealed batch and calls executeBatch. The contract re-derives the merkle root on-chain and rejects anything that is not the revealed batch, then swaps in committed order. The ordering was fixed at step 3, so there was never a window to front-run.

Why batched, and why 48 bytes matters

Threshold-encrypted mempools are not new. The design choice that makes Peal practical is batching the reveal.

per-transactionper-epochbatched (Peal)
committee trafficone share per tx, O(n·B)one key per epochone 48-byte value per operator, O(n)
unincluded txsstay privateexposed at epoch key dropstay private
slot / epoch bindingnonerequirednone

The whole batch of 64 opens on one 48-byte value per operator. Committee traffic is O(n) in the number of operators, not O(n·B) in operators times batch size. The KEM header and the operator shares are both exactly 48 bytes, enforced and tested. That fixed, tiny footprint is what keeps the committee cheap and the reveal fast.

You do not have to trust the reveal: verify it yourself

Every condition has a dashboard at #/condition/<id> showing the sealed hash of each slot before the reveal and the plaintext after, the merkle root, and the per-operator pairing checks. The reveal is designed to be checkable without trusting the coordinator.

verified  The dashboard has a recompute it here button. It rebuilds the merkle root in your browser from all 64 revealed plaintexts, using the exact rule the coordinator uses:

leaf   = sha256(position_le_u32 || payload)
parent = sha256(left || right)     // an odd node is promoted unchanged
root   = fold the leaves in position order

If the recomputed root equals the published one, every revealed plaintext is committed and the reveal was not tampered with. This was checked against a live reveal: the browser recompute matched the coordinator's published root 9dcb6315…7918 exactly. The batch json is downloadable too, so you can run the same check offline.

The two hashes on the dashboard play different roles. The sealed ct hash (before) is sha256(ciphertext), the commitment published on-chain before the batch opens, which is what proves the order was fixed ahead of the cue. The merkle root (after) commits to the revealed plaintexts. The ct hash cannot be recomputed from a plaintext alone (encryption is randomized), so the client-side check is on the merkle root, which is the part that proves the reveal is faithful.

The comparison, one run

A representative run recorded earlier in the build: a $50k swap on the public lane was pushed to its exact 0.5% slippage floor and the searcher took $212.99 on-chain, while the same swap sealed through Peal settled at the full quote with the searcher taking $0. The demo shows both lanes side by side with clickable transaction links, and reports one figure for the loss so the number you see taken from the public lane is the same number the peal lane protects.

Realistic pricing and a safety cap

Before each swap the relayer resets both pools to identical reserves, so the only difference between the lanes is the sandwich, never independent drift. The ETH reserve is set from the live CoinGecko ETH/USD price (cached 60 s, with a fallback), while the USDC-side depth stays constant, so the rate and the dollar values on the page track the real market and the sandwich behaviour stays the same at any price. The demo caps a single swap at $100k of value (USDC one to one, ETH via the live price) so no outsized trade can drain the reset pools.

Hard decisions and trade-offs

  • Same adversary on both lanes. The searcher is the block builder in both worlds. Anything less would be a strawman: the point is that encryption alone closes the gap, with the builder held constant.
  • Settlement in TypeScript, not in the coordinator. Keeping the settler as a separate viem service means the audited devnet coordinator never changed, so the mempool work carries low regression risk.
  • Deterministic order encoding, not JSON. The sealed payload is a fixed byte layout the contract can decode, so the on-chain leaf recomputation matches the reveal byte for byte.
  • Latency is not crypto. Pre-decrypt is pipelined at freeze and combine plus finalize is about 35 ms at batch 64. The old 30-second wait was round length and poll intervals, which is why the cue is now 5 seconds.
  • Pools reset per swap. That makes pool depth a free dial for tuning how small a trade gets sandwiched, and it guarantees a fair, drift-free comparison every time.

What is next

  • Decentralize the committee: a dealerless setup (distributed key generation / powers of tau) so no single party holds the keying material.
  • Operators that verify the cue themselves, and a coordinator-to-client reveal push (SSE) to shave the last poll intervals.
  • Chain-level integration: the same engine behind a sequencer or RPC, rather than an app-level lane.
